02 · Dónde encaja gentle-ai

No compite con Multica. Vive una capa más abajo.

Capa 4 Arquitecto (tú)Define contratos, CONSTITUTION
Capa 3 RepoPLAN.md, src/domain/, tests/contracts/
Capa 2 MulticaOrquesta MACRO — features paralelas en Kanban
Capa 1 opencodeRunner que Multica lanza por issue
Capa 0 gentle-ai overlayDentro de opencode — orquesta MICRO (fases SDD por feature)

Multica orquesta features. gentle-ai orquesta fases. No se pisan — se complementan.

04 · Lo que sí me dio

Razonamiento articulado, no solo código.

Cada agente persistió una memoria estructurada en Engram. Ejemplo del Dev Motor:

Engram memory · type: feature · project: petdesk-v2

What: Implemented StatsEngine in src/engine/stats.ts
Why: Issue ZEN-14 required a stats engine for v3
Where: src/engine/stats.ts, tests/engine.stats.test.ts
Learned: occupancyRate requires intersecting
[rangeStart, rangeEnd) with each day's
9:00–18:00 UTC window for partial days.
Tops need deterministic tie-breaking by id.

Eso es razonamiento articulado. El agente entendió un edge-case sutil y lo dejó escrito como insight reusable. v2 no producía esto.

05 · Lo que cuesta

+60% costo. +92% tiempo. Sin SDD.

El hallazgo no-obvio

"SDD no se activó en ninguno de los 3 issues. El orquestador decidió ir directo cada vez. El +60% costo es el overhead del overlay base — no del flujo SDD."

gentle-ai te factura el overhead del overlay aunque no uses SDD. Skill-registry + Engram + plugins + system prompts custom inyectados en cada call.

06 · El veredicto honesto

No es marketing. Es trade-off.

NO si

  • Tu prioridad es velocidad o costo mínimo
  • Trabajas solo y no necesitas audit trail
  • Features chicas y mecánicas
  • No aprovechas memoria cross-session

SÍ si

  • Equipo de 2+ devs compartiendo decisiones
  • Compliance / auditoría estricta
  • Pipeline de features grandes con decisiones de diseño
  • Querés que el agente articule meta-razonamiento
